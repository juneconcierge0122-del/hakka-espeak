# 論文閱讀筆記

## 📄 Paper 1: SHANKS (2510.06917)

### 1. 一句話

這群人讓語音 AI 學會「一邊聽你講話一邊在腦子裡偷偷想」，就像你跟朋友聊天時，對方話還沒講完你就已經在心裡準備好要怎麼回了。

**一句話講完：** 他們提出 SHANKS 框架，讓語音語言模型（SLM）在用戶還在說話時就能進行無聲的 chain-of-thought 推理，實現「邊聽邊想」。

---

### 2. 📐 關鍵公式與方法步驟

SHANKS 的核心不在單一數學公式，而在一個精巧的 **時序交錯推理架構**。

**【核心機制：Chunked Streaming Inference】**

用戶語音被切成固定長度 `t_chunk` 秒的片段：

```
S = {S₁, S₂, ..., Sₙ}
```

每收到一個語音片段 Sᵢ，模型就生成一段「無聲思考」Rᵢ：

```
Rᵢ = SLM(S₁, R₁, S₂, R₂, ..., Sᵢ₋₁, Rᵢ₋₁, Sᵢ)
```

**【直覺】** 就像你在聽一個人描述數學題，每聽 4 秒你就在腦子裡更新一次「目前知道什麼、可以算什麼」。

**【時間約束】**

每個思考片段 Rᵢ 的 token 數量上限：

```
|Rᵢ| ≤ t_chunk × n_tps
```

其中 `n_tps` 是模型每秒能生成的 token 數。

**【直覺】** 你的「想的速度」不能超過「對方說下一句話的時間」，否則就來不及跟上了。

以 `t_chunk = 4.0s`、A100 GPU 為例，模型有約 320 tokens 的思考預算。

**【訓練損失】**

標準語言建模交叉熵損失：

```
L = -Σᵢ log P(Rᵢ | S₁, R₁, ..., Sᵢ) - log P(O | S₁, R₁, ..., Sₙ, Rₙ)
```

訓練模型預測：給定 {S₁} 預測 R₁，給定 {S₁, R₁, S₂} 預測 R₂⋯最後預測輸出回應 O。

**【情境 1 — 中斷機制】**

當模型在思考中發現用戶犯錯，生成特殊 token `[INTERRUPT]`：

```
如果 Rₖ 包含 [INTERRUPT] → 模型中斷用戶，輸出糾正回應 O
```

**評估指標：**
- **中斷率 (Interrupt ratio)** = 被中斷的實例數 / 總實例數
- **有效中斷率 (Valid interrupt ratio)** = 有效中斷數 / 總中斷數
- **中斷延遲** = `t_interrupt - t_error`（中斷時間 vs 第一個錯誤出現時間）

**【情境 2 — 邊聽邊呼叫工具】**

在旅遊對話場景中，用戶話還沒說完，模型就已經開始呼叫 Booking.com API：

```
用戶: "幫我查杭州到首爾 12 月 10 號最便宜的機票，還有首爾機場附近的租車..."
                                    ↑ 這裡模型就已經開始搜機場代碼了
```

**評估指標：**
- **Early call accuracy**：用戶還在說話時就成功呼叫的 API 比例
- **Late call accuracy**：用戶說完後才呼叫的 API 比例
- **Latency**：用戶說完後模型還需要生成多少 token

---

### 3. 技術優點與不足

**🟢 聰明之處：**

1. **時間差利用極為巧妙**：用戶說第 i+1 段話的時間，剛好給模型思考第 i 段的空間。零額外延遲。
2. **通用框架**：既適用於端到端 SLM（Qwen-omni），也適用於級聯式（Whisper + LLM），不挑架構。
3. **SHANKS + Call-after-listen 組合策略**：聽的時候先做能做的 API call，聽完再補剩下的，call accuracy 從 71.3% 提升到 90.0%，延遲從 313 tokens 降到 117 tokens（降 62.3%）。
4. **推理能力可插拔**：Cascade 版用更強的 text LLM（Qwen-2.5-7B-Instruct），有效中斷率從 63.9% 提升到 78.3%，說明推理能力是可以獨立升級的。

**🔴 潛在不足：**

1. **固定 chunk 大小的限制**（作者有提到）：`t_chunk` 固定導致思考永遠落後語音至少 `t_chunk` 秒。如果用戶的話很短（< 4 秒），根本來不及「邊聽邊想」。
2. **訓練數據全靠 GPT-4o 生成**（作者輕描淡寫）：thinking chunks 和中斷回應都是 GPT-4o 產的，品質天花板就是 GPT-4o。更關鍵的是，用 transcript 而非真實語音來生成訓練數據，忽略了語音中的猶豫、停頓等重要信號。
3. **SHANKS 單獨使用時 tool call 成功率偏低**（作者有提到但輕描淡寫）：單獨 SHANKS-E2E 的 success rate 只有 35.2%，因為失敗的 API call 很少被重試。必須搭配 call-after-listen 才實用。
4. **中斷的正確率仍不夠高**（作者沒特別強調）：correct subset 上也有 24.9% 的中斷率，意味著大約每 4 次正確解答就會被誤中斷一次，在教育場景中這可能很惱人。
5. **沒有真實用戶實驗**：所有用戶語音都是 TTS 合成的，真人說話的不確定性（口吃、跑題、口音）完全沒被測試。
6. **計算成本大增**（作者有提到）：在用戶說話期間持續推理，即使很多推理最後用不上，inference compute 也大幅增加。

---

### 4. 啟發與延伸

- **客語 ASR 場景**：如果要做客語語音助手，SHANKS 的 chunked thinking 架構可以讓模型在聽客語時就開始做語言理解和翻譯準備，特別是客語常有長句，很適合這種邊聽邊想的設計
- **動態 chunk size**：固定 4 秒太粗糙。可以用 VAD（語音活動偵測）或語意斷點來做 adaptive chunking
- **與 STITCH 結合**：作者自己也提到，邊聽邊想 + 邊說邊想可以組合，形成「全程思考」的 SLM
- **多語言 code-switching 場景**：臺灣常見國客混用，模型在聽的過程中判斷語言切換並提前準備對應處理
- **降低誤中斷率**：可引入「信心度門檻」，只有當模型對錯誤的判斷信心超過某閾值時才中斷

## 📄 Paper 2: STITCH (2507.15375)

### 1. 一句話

這群人讓語音 AI 學會「一邊說話一邊在腦子裡想下一步」，就像你演講時嘴巴在講第一段，腦子已經在組織第二段了。

**一句話講完：** 他們提出 STITCH 框架，讓語音語言模型在播放回應語音的同時穿插生成無聲推理 token，實現「邊說邊想」且不增加延遲。

---

### 2. 📐 關鍵公式與方法步驟

**【核心觀察：音訊播放時間 >> token 生成時間】**

```
t_chunk ≈ 2 秒（26 個 speech tokens 對應的音訊時長）
生成 26 speech + 13 text tokens 的時間 ≈ 39/80 ≈ 0.49 秒
```

**【直覺】** 模型吐出一小段語音需要 0.49 秒，但這段語音要播 2 秒。中間空出來的 ~1.5 秒就是「偷偷想事情」的時間。

**剩餘可用 reasoning token 數：**

```
N_reason ≤ n_tps × t_chunk - (N_text + N_speech)
        = 80 × 2 - (13 + 26) 
        = 121 tokens
```

作者保守選 `N_reason = 100`。

**【三種生成管線的比較】**

**(a) GLM-4-Voice 原版**（無推理能力）：
```
[t₁ ◦ s₁ ◦ t₂ ◦ s₂ ◦ ...]
```
直接交替生成 text tokens 和 speech tokens。

**(b) TBS（Think Before Speaking）— trivial baseline**：
```
[z ◦ t₁ ◦ s₁ ◦ t₂ ◦ s₂ ◦ ...]
```
先想完整個 CoT reasoning z，再開始說。問題：z 可以任意長，延遲不可控。

**(c) STITCH-R（Reasoning First）**：
```
[z₁ ◦ t₁ ◦ s₁ ◦ z₂ ◦ t₂ ◦ s₂ ◦ ...]
```
把完整推理切成 N_reason 大小的片段，穿插在 text-speech 之間。First packet latency = N_reason + N_text + N_speech。

**(d) STITCH-S（Speaking First）**：
```
[t₁ ◦ s₁ ◦ z₁ ◦ t₂ ◦ s₂ ◦ z₂ ◦ ...]
```
先說再想！First packet latency = N_text + N_speech，**和沒有推理的 baseline 完全一樣**。

**【訓練】**

從 DTBS 構造訓練數據。每個實例 `(x, z, y)`：
- `x`：用戶語音 token 序列
- `z`：完整 reasoning text
- `y = [t₁ ◦ s₁ ◦ t₂ ◦ s₂ ◦ ...]`：交替的 text-speech tokens

STITCH-R 的訓練目標就是切好的交錯序列：
```
L = -Σ log P(z₁ ◦ t₁ ◦ s₁ ◦ z₂ ◦ t₂ ◦ s₂ ◦ ... | x)
```

**過濾條件**：如果 reasoning chunks 數 > text chunks 數（想的比說的慢），直接丟棄該訓練樣本。

**【推理時可調 N'_reason】**

推理時可以把 N'_reason 從訓練時的 100 調到 60-100，靠強制插入 `[EOPR]` 來截斷。N'_reason ≥ 80 時準確率恢復到 N_reason=100 的 90%。

---

### 3. 技術優點與不足

**🟢 聰明之處：**

1. **STITCH-S 零額外延遲**：這是最驚豔的結果。跟不會推理的 baseline 延遲一樣，但數學準確率高 15%+（GSM8K 上從 35.73% 到 56.72%）。因為模型學會了第一段回應通常是重述問題，不需要推理也能說。
2. **利用「音訊播放空檔」的洞察極為優雅**：不需要任何額外硬體或架構改動，純粹是聰明地安排 token 生成順序。
3. **N_reason 推理時可調**：訓練一個模型，部署時可以根據硬體速度調整推理 token 數，實用性很高。
4. **外部 reasoning augmentation**：可以用更強的 LLM 來生成 reasoning，然後餵給 STITCH-R 來說。模型真的有在用推理內容，不是裝樣子。
5. **非推理任務不掉分**：在知識問答和對話任務上，STITCH 和不推理的 baseline 表現一樣好，說明推理能力的加入沒有副作用。

**🔴 潛在不足：**

1. **只在 GLM-4-Voice 上驗證**（作者有提到理由但仍是限制）：沒有在 thinker-talker 架構（如 Qwen-Omni）上實驗，主要因為 tokenizer 未公開。泛化性存疑。
2. **訓練數據量巨大（~400K）且依賴 GPT-4o**：reasoning 全部由 GPT-4o 生成，訓練成本高。
3. **推理比說話慢就丟棄樣本**（作者提到但沒分析影響）：這個過濾條件意味著需要長推理的困難問題可能被系統性地排除。GSM8K 這種難題上的 performance drop 可能跟此有關。
4. **語音品質評估不夠全面**（作者沒特別提）：只用 UTMOSv2 自動評估，沒有大規模人類聽感測試。穿插 reasoning 是否會影響語音的自然度和韻律連貫性，需要更多驗證。
5. **LoRA fine-tuning 完全失敗**（附錄提到）：只有 full fine-tuning 才有效，意味著計算門檻很高。

**🔴 作者沒有提到的：**

6. **reasoning 與 speech 的語義「斷裂」問題**：reasoning chunk 會在任意位置被截斷（到了 N_reason 就強制 `[EOPR]`），這可能導致推理邏輯不完整。雖然結果顯示影響不大，但在更複雜的推理任務上可能成為瓶頸。
7. **第一段回應「重述問題」的假設不一定成立**：STITCH-S 能成功是因為模型學會第一段先重述。但如果應用場景需要模型立即給出實質內容（如即時翻譯），這個假設就崩了。

---

### 4. 啟發與延伸

- **跟 SHANKS 組合**：SHANKS 邊聽邊想 + STITCH 邊說邊想 = 全程思考的 SLM。作者在兩篇論文中都互相引用了對方。
- **客語語音助手**：對客語 ASR 系統，回應時邊說邊想可以讓模型在講國語回應的同時，推理下一段是否需要切換到客語
- **Adaptive N_reason**：不同 chunk 的推理需求不同。可以用 reinforcement learning 讓模型自己決定每段要想多少
- **多模態延伸**：同樣的「利用播放空檔」概念可以用在影片生成——生成一幀的同時推理下一幀的佈局
- **Streaming 場景的知識更新**：如果 reasoning augmentation model 可以即時接收外部知識更新，STITCH 就能在對話過程中動態整合新資訊

## 🔗 兩篇論文的關係

這兩篇是**同一個第一作者（江承翰，NTU + Microsoft）的姊妹作**：

| | SHANKS | STITCH |
|---|---|---|
| **何時思考** | 用戶說話時 | 模型說話時 |
| **核心洞察** | 用戶說下一段的時間 = 模型思考時間 | 音訊播放時間 >> token 生成時間 |
| **延遲影響** | 零（利用用戶說話空檔） | 零（STITCH-S）或極低（STITCH-R） |
| **發表狀態** | Work in Progress | ICLR 2026 |
| **backbone** | Qwen-omni / Qwen-2.5-7B-Instruct | GLM-4-Voice-9B |

兩者可以組合成「全程思考 SLM」：邊聽邊想 → 想完開始說 → 邊說邊想。

---

**適合誰讀：** 對語音語言模型（SLM）、即時對話系統、或低延遲推理有興趣的研究生。如果你在做客語 ASR，SHANKS 的 chunked streaming 架構和 STITCH 的邊說邊想概念都是值得參考的設計模式。
